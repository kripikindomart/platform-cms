## 🏗️ Final Database Structure

### PUBLIC SCHEMA (Shared/Global)

**Purpose**: Data yang di-share across semua tenant

```
public
├── Core Tables
│   ├── users (23 rows)              ← ALL users (shared)
│   ├── tenants (1 row)              ← Tenant registry
│   └── modules (X rows)             ← Available modules
│
├── Module Management (CLI Generator)
│   ├── module_permissions           ← Permission templates
│   ├── module_fields                ← Field definitions
│   ├── module_migrations            ← Migration tracking
│   ├── generated_modules            ← Generation history
│   └── generation_history           ← Generation log
│
├── Configuration
│   ├── system_settings              ← Global settings
│   ├── field_validations            ← Validation rules
│   └── tenant_modules               ← Module registry per tenant
│
└── Backups
    └── backup_tenant_users_20260716 ← Backup table (can be deleted)
```

### TENANT_DEMO_COMPANY SCHEMA (Isolated)

**Purpose**: Data yang isolated per tenant

```
tenant_demo_company
├── Auth & Authorization
│   ├── roles (4 rows)               ← Tenant-specific roles
│   ├── permissions (14 rows)        ← Tenant-specific permissions
│   ├── user_roles (1 row)           ← FK to public.users
│   └── role_permissions (14 rows)   ← Role-permission mapping
│
├── Security & Audit
│   ├── sessions (0 rows)            ← User sessions
│   ├── password_resets (0 rows)     ← Password reset tokens
│   └── audit_logs (0 rows)          ← Activity logs
│
├── Navigation
│   └── (menus will be here if generated)
│
└── Business Data (Generated by CLI)
    ├── categories                   ← Generated module
    ├── tags                         ← Generated module
    └── (products, orders, etc)      ← Future generated modules
```

## 🔄 Data Flow

### 1. User Login
```
Request:
  POST /api/auth/login
  Header: X-Tenant-Slug: demo_company
  Body: { email, password }

Backend:
  1. Query: SELECT * FROM public.users WHERE email = ?
  2. Verify password
  3. Validate tenant exists in public.tenants
  4. Generate JWT with { userId, tenantId }
  
Response:
  { access_token, user }
```

### 2. Protected Request
```
Request:
  GET /api/roles
  Header: Authorization: Bearer <token>
  Header: X-Tenant-Slug: demo_company

Backend:
  1. JWT Strategy validates token
  2. TenantMiddleware sets context: tenant_demo_company
  3. UsersService.findByIdWithRoles(userId)
     a. Query: SELECT * FROM public.users WHERE id = ?
     b. Query: SELECT * FROM tenant_demo_company.user_roles WHERE user_id = ?
     c. Query: SELECT * FROM tenant_demo_company.roles WHERE id IN (...)
     d. Query: SELECT * FROM tenant_demo_company.permissions WHERE id IN (...)
  4. CASL checks permissions
  5. Query: SELECT * FROM tenant_demo_company.roles
  
Response:
  [{ id, name, display_name, ... }]
```

## 📊 Query Pattern

### ✅ ALWAYS Query Public (Shared)
```sql
-- Users (authentication)
SELECT * FROM public.users WHERE email = 'user@example.com';

-- Tenants (validation)
SELECT * FROM public.tenants WHERE slug = 'demo_company';

-- Modules (available modules)
SELECT * FROM public.modules WHERE is_active = true;
```

### ✅ ALWAYS Query Tenant (Isolated)
```sql
-- Set search_path first (handled by withTenantSchema)
SET search_path TO "tenant_demo_company", public;

-- Roles
SELECT * FROM roles WHERE deleted_at IS NULL;

-- User roles with permissions
SELECT r.*, p.*
FROM user_roles ur
JOIN roles r ON ur.role_id = r.id
JOIN role_permissions rp ON r.id = rp.role_id
JOIN permissions p ON rp.permission_id = p.id
WHERE ur.user_id = ?;
```

## 🔗 Foreign Key Relationships

### Cross-Schema FK (Tenant → Public)
```sql
-- tenant_demo_company.user_roles references public.users
ALTER TABLE tenant_demo_company.user_roles
  ADD CONSTRAINT user_roles_user_id_fkey
  FOREIGN KEY (user_id) REFERENCES public.users(id) ON DELETE CASCADE;

-- Same pattern for:
-- - sessions.user_id → public.users.id
-- - audit_logs.user_id → public.users.id
-- - roles.created_by → public.users.id
-- - etc (total 17 FK constraints)
```

### Same-Schema FK (Tenant → Tenant)
```sql
-- tenant_demo_company.user_roles references tenant_demo_company.roles
ALTER TABLE tenant_demo_company.user_roles
  ADD CONSTRAINT user_roles_role_id_fkey
  FOREIGN KEY (role_id) REFERENCES tenant_demo_company.roles(id) ON DELETE CASCADE;
```

## 🎯 Benefits of This Architecture

### 1. **Strong Isolation**
- Tenant A cannot query Tenant B's data
- DROP SCHEMA tenant_a CASCADE = complete tenant removal
- Per-tenant backup: `pg_dump -n tenant_demo_company`

### 2. **Flexible Roles/Permissions**
- Each tenant can have completely different roles
- Tenant A: `admin`, `editor`, `viewer`
- Tenant B: `manager`, `cashier`, `accountant`

### 3. **Scalable Module Generation**
```bash
# Generate 'products' module for demo_company
npm run generate:module products --tenant demo_company

# Creates:
# - tenant_demo_company.products (table)
# - src/modules/products/* (code)
```

### 4. **Cross-Tenant Users**
```
User: john@email.com (ID: 5)

public.users:
  id: 5, email: john@email.com, name: John Doe

tenant_demo_company.user_roles:
  user_id: 5, role_id: 2 (editor)

tenant_company_b.user_roles:
  user_id: 5, role_id: 1 (admin)
```

Same user, different roles per tenant!

### 5. **Clear Code Separation**
```typescript
// Users Repository - ALWAYS query public
class UsersRepository {
  async findByEmail(email: string) {
    return this.db.select().from(users); // public.users
  }
}

// Roles Repository - ALWAYS query tenant
class RolesRepository extends BaseRepository {
  async findAll() {
    return this.withTenantSchema(() => {
      return this.db.select().from(roles); // tenant_xxx.roles
    });
  }
}
```

## 🚫 Anti-Patterns to Avoid

### ❌ Querying Wrong Schema
```typescript
// WRONG: Query tenant schema for users
const user = await db.select()
  .from(tenantSchema.users) // ❌ Does not exist!
  .where(eq(tenantSchema.users.email, email));

// RIGHT: Query public schema for users
const user = await db.select()
  .from(publicSchema.users) // ✅ Correct
  .where(eq(publicSchema.users.email, email));
```

### ❌ Mixing Public and Tenant Data
```typescript
// WRONG: Join across schemas without context
const data = await db.select()
  .from(publicSchema.users)
  .join(tenantSchema.roles, ...); // ❌ Ambiguous!

// RIGHT: Load separately
const user = await usersRepo.findById(userId); // public
const roles = await rolesRepo.getUserRoles(userId); // tenant (via context)
```

## 📈 Scaling Considerations

### When to Refactor to Row-Level Security (RLS)

Consider moving from **schema-per-tenant** to **RLS** if:

1. **500+ tenants** - Too many schemas
2. **Cross-tenant analytics** needed
3. **Cloud hosting** with schema limits (AWS RDS, etc)
4. **Simpler migrations** - One schema to migrate

### Current Limits
- PostgreSQL handles 1000+ schemas well
- Migration complexity: Run on each tenant schema
- Backup time: Proportional to tenant count

## 🧪 Testing Queries

### Test User Access
```sql
-- Check user exists
SELECT id, email, name FROM public.users WHERE email = 'admin@platform.com';

-- Check user has roles in tenant
SELECT 
  u.email,
  r.name as role_name,
  COUNT(rp.permission_id) as permission_count
FROM public.users u
JOIN tenant_demo_company.user_roles ur ON u.id = ur.user_id
JOIN tenant_demo_company.roles r ON ur.role_id = r.id
LEFT JOIN tenant_demo_company.role_permissions rp ON r.id = rp.role_id
WHERE u.email = 'admin@platform.com'
GROUP BY u.email, r.name;
```

### Test FK Constraints
```sql
-- All FK from tenant to public.users
SELECT
  tc.table_name,
  kcu.column_name,
  ccu.table_schema AS foreign_schema,
  ccu.table_name AS foreign_table
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage ccu ON tc.constraint_name = ccu.constraint_name
WHERE tc.constraint_type = 'FOREIGN KEY'
  AND tc.table_schema = 'tenant_demo_company'
  AND ccu.table_schema = 'public'
  AND ccu.table_name = 'users';
```
